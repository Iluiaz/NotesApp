ВВЕДЕНИЕ
Целью лабораторной работы №11 является разработка простого мобильного приложения для хранения заметок
на платформе Android и подготовка модульных тестов для проверки основной функциональности.
В рамках работы разработано приложение Notes App на Kotlin. Приложение поддерживает создание заметки
(заголовок + текст), редактирование, удаление, просмотр списка заметок и локальное хранение данных.
В текущей реализации данные сохраняются локально в SharedPreferences в виде JSON-структуры.

ЧАСТЬ 1: АРХИТЕКТУРА ПРИЛОЖЕНИЯ
1.1 Технологический стек
Язык: Kotlin 1.9. Min SDK: API 24. Target / Compile SDK: API 36.
UI: XML Layouts, RecyclerView, Material Components, ViewBinding.
Локальное хранение: SharedPreferences + JSON.
Архитектура: Activity + Repository + Storage слой.
Тестирование: JUnit 4, Robolectric, AndroidX Test.

1.2 Структура проекта
com.example.notesapp/
  model/
    Note.kt                         # модель заметки
  data/
    NoteStorage.kt                  # интерфейс хранилища
    SharedPreferencesNoteStorage.kt # локальное хранилище
    NoteJsonSerializer.kt           # сериализация JSON
    NoteRepository.kt               # CRUD-логика
  ui/
    NotesAdapter.kt                 # RecyclerView Adapter
  MainActivity.kt                   # главный экран и диалоги

ЧАСТЬ 2: ОСНОВНЫЕ КОМПОНЕНТЫ
2.1 Модель Note
Модель Note содержит поля: id, title, content, updatedAt.
Поле id используется для идентификации заметки, updatedAt — для сортировки по времени изменения.

2.2 Репозиторий и локальное хранение
Класс NoteRepository инкапсулирует операции getAll(), create(), update(), delete().
SharedPreferencesNoteStorage отвечает за чтение и запись JSON-строки в локальное хранилище.
NoteJsonSerializer преобразует список заметок в JSON и обратно.
Такое разделение упрощает тестирование и исключает жёсткую привязку логики к UI.

2.3 Пользовательский интерфейс
Главный экран реализован в MainActivity и содержит RecyclerView со списком заметок.
Кнопка FloatingActionButton открывает диалог создания заметки.
Для каждой заметки доступны действия Edit и Delete.
Редактирование выполняется через тот же диалог с предзаполненными полями.
Удаление выполняется после подтверждения пользователя.

ЧАСТЬ 3: ОПИСАНИЕ ФУНКЦИОНАЛА
3.1 Просмотр списка заметок
После запуска отображается список заметок, отсортированный по updatedAt по убыванию.
Если заметок нет, пользователю показывается пустое состояние с подсказкой создать первую заметку.

3.2 Создание заметки
Диалог содержит два поля: Title и Content.
Заголовок обязателен. После сохранения новая заметка отображается в начале списка.

3.3 Редактирование заметки
При выборе Edit открывается диалог с текущими данными заметки.
После сохранения заголовок, текст и время обновления изменяются.

3.4 Удаление заметки
Удаление производится через кнопку Delete и подтверждающий AlertDialog.
После удаления заметка исключается из списка и из локального хранилища.

3.5 Локальное хранение
Данные сохраняются локально на устройстве в SharedPreferences.
При повторном запуске приложения сохранённые заметки загружаются автоматически.

ЧАСТЬ 4: МОДУЛЬНОЕ ТЕСТИРОВАНИЕ (JUNIT)
4.1 Стратегия тестирования
Для проверки бизнес-логики подготовлены модульные тесты для репозитория и сериализатора.
Тесты не зависят от интерфейса и запускаются изолированно на уровне слоя данных.

4.2 Реализованные тесты
Файл: app/src/test/java/com/example/notesapp/data/NoteRepositoryTest.kt
- create_savesNoteAndReturnsIt — создание заметки и сохранение в хранилище;
- create_emptyTitleThrows — проверка валидации пустого заголовка;
- update_changesExistingNote — обновление существующей заметки;
- delete_removesNoteFromStorage — удаление заметки;
- getAll_loadsSavedNotes — загрузка ранее сохранённых заметок.

Файл: app/src/test/java/com/example/notesapp/data/NoteJsonSerializerTest.kt
- serializeAndDeserialize_roundTripPreservesNotes — корректная сериализация и десериализация списка.

4.3 Подготовка окружения для запуска тестов
Для запуска модульных тестов используется задача Gradle: ./gradlew test
В текущей среде разработки отсутствует локально настроенный Gradle wrapper и Android Build Tools,
поэтому фактический прогон выполняется после открытия проекта в Android Studio и выполнения Gradle Sync.

4.4 Ожидаемый результат прогона
После синхронизации проекта в Android Studio должны успешно выполняться 6 модульных тестов:
5 тестов репозитория + 1 тест сериализатора.
Скриншот результата прогона рекомендуется вставить в отчёт как подтверждение выполнения задания.
