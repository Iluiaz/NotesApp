ВВЕДЕНИЕ
Лабораторная работа №12 посвящена UI-тестированию мобильного приложения Notes App
с использованием Espresso, а также подготовке профилирования потребления ресурсов
через Android Profiler. Основная логика приложения была реализована в ЛР №11.
На данном этапе внимание сосредоточено на пользовательских сценариях и анализе ресурсов устройства.

ЧАСТЬ 1: UI-ТЕСТИРОВАНИЕ (ESPRESSO)
1.1 Фреймворк Espresso
Espresso — инструмент автоматизированного тестирования интерфейсов Android-приложений.
Он позволяет находить элементы интерфейса, выполнять действия пользователя и проверять результат.
Для Notes App Espresso используется для проверки ключевых операций с заметками через UI.

1.2 Подключённые зависимости
androidTestImplementation("androidx.test.ext:junit:1.2.1")
androidTestImplementation("androidx.test.espresso:espresso-core:3.6.1")
androidTestImplementation("androidx.test.espresso:espresso-contrib:3.6.1")
androidTestImplementation("androidx.test:rules:1.6.1")

1.3 Реализованные UI-тесты
Файл: app/src/androidTest/java/com/example/notesapp/MainActivityTest.kt
UI-01 createNote_displaysInList — создание заметки через интерфейс и проверка её отображения.
UI-02 editNote_updatesDisplayedContent — редактирование заметки и проверка обновлённых данных.
UI-03 deleteNote_removesDisplayedItem — удаление заметки и проверка пустого состояния списка.

Дополнительный helper:
RecyclerViewChildClickAction.kt — вспомогательное действие для нажатия на кнопки Edit/Delete
внутри элементов RecyclerView.

1.4 Описание сценариев
UI-01 проверяет полный цикл создания заметки: открыть диалог, заполнить Title и Content,
нажать Save и убедиться, что заметка появилась в списке.
UI-02 проверяет редактирование существующей заметки: открыть запись, изменить значения,
сохранить и убедиться, что в списке отображаются новые данные.
UI-03 проверяет удаление: создать тестовую заметку, нажать Delete, подтвердить удаление
и убедиться, что элемент исчез из списка, а пустое состояние снова отображается.

1.5 Запуск UI-тестов
Для запуска UI-тестов используется задача Gradle: ./gradlew connectedAndroidTest
Тесты выполняются на эмуляторе или подключённом Android-устройстве.
В текущей среде подготовки отчёта отсутствует полностью настроенный Gradle wrapper,
поэтому фактический запуск выполняется после открытия проекта в Android Studio.

1.6 Что необходимо вставить в отчёт после запуска
- скриншот успешного прогона Espresso-тестов;
- скриншот пустого экрана приложения;
- скриншот создания заметки;
- скриншот списка с сохранённой заметкой;
- скриншот окна удаления / подтверждения удаления.

ЧАСТЬ 2: ПРОФИЛИРОВАНИЕ (ANDROID PROFILER)
2.1 Методика профилирования
Профилирование выполняется в Android Studio с помощью Android Profiler.
Рекомендуется использовать эмулятор Pixel с API 36 или физическое устройство в режиме Debug.
Необходимо фиксировать показатели CPU, Memory и при необходимости Energy во время
основных пользовательских сценариев работы приложения.

2.2 Сценарии для профилирования
P-01 Запуск приложения и первичная загрузка списка заметок.
P-02 Создание новой заметки через диалог.
P-03 Редактирование существующей заметки.
P-04 Удаление заметки.
P-05 Повторный запуск приложения и загрузка ранее сохранённых данных.

2.3 Что следует снять из Android Profiler
- график CPU во время запуска и CRUD-операций;
- график Memory во время работы со списком заметок;
- скриншоты профайлера для каждого сценария P-01 ... P-05.

2.4 Наблюдения по архитектуре приложения
Приложение не использует сеть и не обращается к удалённым API.
Основная нагрузка приходится на операции чтения/записи в SharedPreferences, сериализацию JSON
и обновление RecyclerView после CRUD-действий.
Поэтому в профилировании следует обратить внимание прежде всего на кратковременные пики CPU
во время сохранения и на потребление памяти при работе со списком заметок.

2.5 Рекомендации по анализу результатов
Если при CRUD-операциях наблюдаются заметные скачки CPU или Memory, следует проверить:
1. частоту полной перерисовки списка;
2. объём сериализуемых данных;
3. работу адаптера и частоту обновления элементов;
4. влияние объёма заметок на скорость локального сохранения.
